## 技术经验分类

### 🐛 Bug修复
#### H1-1: Async/Sync Session 混用
- **问题描述**：API endpoints 中混用 `Session` (sync) 和 `AsyncSession` (async)，导致数据库操作不一致
- **根本原因**：FastAPI 中 `Depends(get_db)` 返回的类型在不同 endpoint 中没有统一约定
- **解决方法**：所有端点统一改为 `AsyncSession` + `select()` 语法，`db.add()` + `await db.commit()` + `await db.refresh()`
- **预防建议**：在 `get_db` generator 中明确声明返回 AsyncSession，或在 Pydantic schema 中约束

#### H1-2: User 模型缺少 is_active 字段 — status 重命名为 is_active
- **问题描述**：旧代码使用 `current_user.status == 1` 判断用户状态，与模型定义不一致
- **解决方法**：将 `status` 重命名为 `is_active`，类型从 `status` 改为 `int (0/1)`
- **预防建议**：Pydantic schema 和 SQLModel 模型之间使用明确的字段映射层（如 `UserResponse`）

#### H1-3: User 模型缺少 role_id 外键
- **问题描述**：创建/更新用户时未正确传递 role_id
- **解决方法**：在 `UserCreate` schema 中添加 role_id 字段，create/update 中正确传递
- **预防建议**：使用 SQLAlchemy relationship 自动加载/设置 role，避免手动管理外键

#### H1-4: users.py 搜索条件 — User.full_name 改为 User.realname
- **问题描述**：查询时使用了 model 中不存在的 `full_name` 属性
- **解决方法**：改为 `User.realname`，并在 update 时做 `full_name` → `realname` 映射
- **预防建议**：在 model 变更时同步更新所有查询/过滤代码，使用 IDE 全局搜索

#### H1-5: Pydantic v2 兼容性 — dict() 改为 model_dump()
- **问题描述**：Pydantic v2 弃用了 `.dict()` 方法，3个 endpoint 报错
- **解决方法**：`model.dict()` → `model.model_dump()`，`model.dict(exclude_unset=True)` → `model.model_dump(exclude_unset=True)`
- **预防建议**：统一 Pydantic 版本，在 CI 中添加 Pydantic v2 兼容性检查

#### H1-6: 状态字段命名统一
- **问题描述**：状态字段在不同文件中使用了 `is_active`、`is_enabled` 等不同命名
- **解决方法**：统一为 `is_active`，并添加类型转换（bool → int）
- **预防建议**：建立命名规范文档，枚举状态字段的标准命名

### ♻️ 重构优化
#### 删除 `_sa_instance_state` 清理
- **优化内容**：`user.__dict__.pop('_sa_instance_state', None)` — 每次返回 user dict 前清理 SQLAlchemy 内部字段
- **优化方向**：避免 JSON 序列化时暴露内部状态
- **效果**：减少 3 处潜在的信息泄露 bug

#### menus.py 子菜单检查改进
- **优化内容**：`select(func.count(Menu.id))` → `select(Menu.id)` + `.scalars().first()`
- **优化方向**：用 exists 模式替代 count，性能更好
- **效果**：找到第一条子菜单即返回，不需要计数

### 📦 依赖变更
#### SQLAlchemy 从 sync 迁移到 async
- **变更原因**：匹配 FastAPI 的异步架构，避免阻塞事件循环
- **注意事项**：`db.query()` → `await db.execute(select())` 是核心变更模式；需要检查所有 endpoint
- **代码模式回顾**：
  - Old: `db.query(Model).filter().first()`
  - New: `result = await db.execute(select(Model).filter()); result.scalars().first()`

## 💡 经验总结
1. **Async/Await 统一化**：FastAPI 项目中所有数据库操作应统一为 async 模式，避免混用 sync/async
2. **Pydantic v2 迁移**：`.dict()` → `.model_dump()` 是最低成本的迁移，但注意所有字段映射逻辑（如 `full_name` → `realname`）需要同步更新
3. **状态字段规范**：布尔状态建议使用 `int (0/1)` 存储（兼容性好），schema 层接收 `bool`，模型层做类型转换
4. **SQLAlchemy 内部字段清理**：直接序列化 `__dict__` 时会带入 `_sa_instance_state`，应清理或使用 `model_dump()`
5. **API 端点命名一致性**：`is_active` vs `is_enabled` → 建立全局常量/枚举管理状态字段
